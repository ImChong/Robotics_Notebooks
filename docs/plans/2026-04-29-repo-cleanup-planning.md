# 仓库冗余清理计划（草案）

> 状态：仅规划，不执行删除/迁移操作
> 
> 日期：2026-04-29

## 1. 目标与范围

本轮仅完成“清理计划设计”，不直接修改知识内容与前端渲染逻辑，不删除任何文件。

目标：
- 识别仓库中的潜在冗余文件与重复内容。
- 明确“保留 / 归档 / 合并 / 删除候选”的判定标准。
- 形成可分批执行的低风险清理路线。

范围：
- 优先覆盖 `docs/`、`sources/`、`wiki/`、`exports/`。
- 暂不涉及大规模前端重构与导出机制改造。

## 2. 清理原则

1. **知识优先**：不为兼容旧展示而牺牲 wiki 结构。
2. **可追溯**：任何删除候选必须可说明“为何可删、是否可恢复”。
3. **分批推进**：先盘点后执行，每批次都可回滚。
4. **避免误删**：带引用关系的文件优先做“迁移/合并”，而非直接删除。
5. **先收敛规范再落地**：先统一命名与目录约定，再做批量清理。

## 3. 冗余类型定义（判定口径）

- **重复版本冗余**：同类文档多版本并存，仅少数仍在使用。
- **失效资产冗余**：前端/导出不再引用的静态资源。
- **结构性冗余**：同一知识点在多个页面重复叙述且无互链分工。
- **迁移残留冗余**：旧目录中的临时稿、备份稿、一次性中间产物。
- **命名冗余**：命名不统一导致语义重复（如 near-duplicate 页面）。

## 4. 分阶段执行计划

### Phase A：资产盘点（Inventory）

输出物：`docs/checklists/repo-cleanup-inventory-v1.md`

任务：
- 建立目录级清单（按 `docs/`、`sources/`、`wiki/`、`exports/` 分组）。
- 标记每个候选文件的用途、最近维护状态、被引用关系。
- 区分“确定冗余”与“待确认冗余”。

### Phase B：依赖与引用分析（Reference Audit）

输出物：`docs/checklists/repo-cleanup-reference-audit-v1.md`

任务：
- 审核 `docs/index.html`、`docs/main.js`、`docs/search-index.json` 对资源引用。
- 审核 wiki 页面互链与来源引用是否完整。
- 对删除候选建立“影响面说明”（页面、链接、导出链路）。

### Phase C：候选分级与处置策略（Triage）

输出物：`docs/checklists/repo-cleanup-triage-v1.md`

分级建议：
- **P0（高置信可清理）**：无引用、无维护、可替代。
- **P1（需迁移后清理）**：有残余引用，需先改链接或合并内容。
- **P2（暂缓）**：仍有潜在价值，先归档不删除。

处置动作：
- 保留（Keep）
- 合并（Merge）
- 归档（Archive）
- 删除（Delete）

### Phase D：小批次执行与回归检查（Execution）

输出物：
- 每批次一个执行记录：`docs/change-log.md`
- 如有必要，补充专项复盘：`docs/plans/`

任务：
- 每次只处理同类型小批次文件。
- 执行后验证链接、索引、图谱统计一致性。
- 不通过验证则立即回滚该批次。

## 5. 验收标准（DoD）

满足以下条件才进入“执行清理”：
- 已完成 Inventory + Reference Audit + Triage 三份清单。
- 每个删除候选都有“引用检查记录 + 风险说明 + 回滚方案”。
- 团队确认首批清理范围（建议从 `docs/checklists/` 历史版本开始）。

## 6. 风险与回滚

主要风险：
- 删除后导致前端页面断链。
- 合并文档时丢失历史上下文。
- 统计数据不同步引发 CI 失败。

回滚策略：
- 以“批次”为单位提交，出现问题按批次 `git revert`。
- 删除前先迁移到归档目录（如 `docs/archive/`）进行过渡。
- 清理后立即执行 lint 与导出校验流程。

## 7. 建议的首批试点（仅建议）

优先从低风险、高收益区域开始：
1. `docs/checklists/` 历史版本清单梳理（保留最新版 + 历史索引）。
2. `docs/plans/` 中已完成且重复的计划稿归档策略。
3. `docs/` 下未被入口页面引用的孤立文档核查。

- 盘点结果会形成可审阅 checklist，再由你决定第一批实际清理项。
